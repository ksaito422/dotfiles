# 汎用コード原則参照

## エラーハンドリング

- [ ] `null` / `nil` / `undefined` が返ってくる可能性のあるすべての箇所を処理しているか
- [ ] 空のコレクション（空配列、空ハッシュ）に対して正しく動作するか
- [ ] 数値の境界値（ゼロ除算、整数オーバーフロー、負の値）を考慮しているか
- [ ] 外部 API やデータベースが応答しない / タイムアウトした場合の処理があるか
  - [ ] 適切にリトライやタイムアウトが設定されているか
- [ ] 例外を握りつぶしていないか（`rescue => e; end` / `catch(() => {})` など）
- [ ] 補足した例外を適切にログに記録しているか
- [ ] 汎用の `Exception` / `Error` ではなく、具体的な例外クラスを捕捉しているか
- [ ] エラーが起きた際にユーザーに適切なフィードバックを返しているか

## コード品質

- [ ] SOLID原則に従っているか
  - 単一責任の原則
    - [ ] クラスや関数が複数の責任を持っていないか
    - [ ] 「〜かつ〜する」という説明が必要な場合は分割を検討する
  - 開放/閉鎖の原則
    - [ ] 既存コードを変更せずに機能を拡張できるか
    - [ ] 多数の `if/else` や `case` で型や種類を判定していないか（ポリモーフィズムを活用できないか）
  - リスコフの置換原則
  - インターフェース分離の原則
  - 依存性逆転の原則
    - [ ] 具体的な実装ではなく抽象（インターフェース）に依存しているか
    - [ ] テストのためにモックに差し替えられるか（依存関係の注入が可能か）
- KISS原則に従っているか
  - [ ] コードはシンプルか
- YAGNI原則に従っているか
  - [ ] 汎用性よりも単純性を求め、今必要なものだけを実装しているか
- DRY原則に従っているか
  - [ ] 重複コードがないか
  - [ ] 共通のロジックは関数やクラスにまとめられているか
- [ ] 命名は明確で周囲のコードベースと一貫しているか？
- [ ] デッドコード、コメントアウトされたコード、放置されたTODOはないか
- [ ] 循環的複雑度(Cyclomatic Complexity）が高すぎないか
  - 分岐やループが多すぎる関数はないか

## テスト品質

- [ ] 実装したメソッドに対するテストが書かれているか
- [ ] テストの実行速度が十分に速くなるように考慮されているか
  - [ ] 定数などをモックしてデータの必要量を最小限にできているか
- [ ] テストのケースと期待値を読むだけで「何をテストしているか」が分かるか
- [ ] 不必要にプライベートメソッドをテストしていないか（基本、テストはパブリックな振る舞いを検証するべき）
- [ ] 正常系だけでなく異常系・境界値・エッジケースがテストされているか
- [ ] テストが独立していて、テスト間で状態が共有されていないか
- [ ] テストが実行順序に依存していないか（各テストが独立して動くか）
- [ ] フィクスチャ / ファクトリが肥大化して、テストの意図が不明確になっていないか
- [ ] テストケースに合わせて書き方を工夫できているか
  - 網羅性を重視する場合や境界値テストで、パラメタライズドテストを用いて可読性を保ちつつ多くのケースをカバーできているか
- [ ] 重要なケースを重点的に書いているか（全てのケースを網羅する必要はない）
- [ ] 削除したコードに対するテストも削除されているか

## コメント

- [ ] コメントが「何をしているか」ではなく「なぜそうしているか」を説明しているか
- [ ] コードを読めば分かることをコメントしていないか（冗長なコメント）
  - 周囲のコードベースに存在しない概念や、コードからは明らかでない理由を説明するためのコメントは有用
- [ ] コメントアウトされたコードが残っていないか（バージョン管理を信頼して削除する）
- [ ] TODO/FIXME コメントが具体的な内容を含んでいるか

## ログとオブザーバビリティ

- [ ] 重要な操作（認証、決済、状態変化）がログに記録されているか
- [ ] ログにパスワード・トークン・個人情報が含まれていないか
- [ ] エラーログに原因を追跡するための十分なコンテキストが含まれているか（ユーザー ID、リクエスト ID など）

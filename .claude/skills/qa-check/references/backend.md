# Ruby/Rails QAチェック項目

## エラーハンドリング

- [ ] `rescue` で捕捉する例外クラスが具体的か（`StandardError` や `Exception` を広く捕捉していないか）
- [ ] 例外を握りつぶしていないか（`rescue => e; end`）
- [ ] 補足した例外をログに記録しているか
- [ ] 外部API / DBがタイムアウトした場合の処理があるか（リトライ・タイムアウト設定）
- [ ] コントローラーで `rescue_from` を使って例外を適切なHTTPステータスに変換しているか
- [ ] リソースの解放が保証されているか
  - [ ] `Tempfile` を `close` & `unlink` しているか（またはブロック形式で自動解放）
  - [ ] `File.open` をブロック形式で使っているか（例外時も `close` が保証される）
  - [ ] `IO` / ストリームを `ensure` で確実に `close` しているか

## コード品質

- [ ] カプセル化が適切か
  - [ ] 内部状態を外部から直接変更できる `attr_writer` / `attr_accessor` を不必要に公開していないか
  - [ ] 外部から変更されるべきでない属性は `private` または読み取り専用（`attr_reader`）にしているか
- [ ] インスタンス生成時の初期化（`initialize`）を不完全な状態で終えていないか
  - [ ] 必要な依存関係や設定値が注入されているか
  - [ ] システム日時やランダム値などの外部状態に依存していないか
- [ ] インスタンス生成時の初期化（`initialize`）時に副作用のある操作をしていないか
  - [ ] DBアクセスや外部API呼び出しなどの重い処理を `initialize` に書いていないか
  - [ ] インスタンス生成時に例外が発生する可能性がある処理を避けているか
  - [ ] 必要な初期化はクラスメソッド（例: `User.create_with_defaults`）やファクトリーメソッドで行うことを検討しているか
- [ ] 副作用を想像できないメソッド名で副作用のある操作をしていないか
  - [ ] `save` / `create` / `update` のような状態変更を伴うメソッドは明示的に命名されているか（例: `generate_report!`、`send_notification!`）
  - [ ] 単純な値の取得や計算を行うメソッドが副作用を持っていないか（例: `user.name` がDBアクセスや外部API呼び出しをしていないか）
  - [ ] 副作用のあるメソッドは末尾に `!` を付けるなど、慣習的な命名規則を守っているか
- [ ] Fat Model / Fat Controller になっていないか
  - [ ] ビジネスロジックがコントローラーに漏れていないか
  - [ ] 複雑なビジネスロジックは PORO / Service Object に切り出しているか
  - [ ] Form Object で複雑なフォーム処理を管理しているか
- [ ] scope が `ActiveRecord::Relation` を返しているか
  - `nil` を返す可能性のある条件分岐を含む場合は `scope` ではなくクラスメソッドを使う
- [ ] コールバックを乱用していないか
  - [ ] `after_save`、`after_create` で複雑な副作用を起こしていないか
  - [ ] テストが難しくなるような隠れたコールバックチェーンがないか
  - [ ] コールバックの代わりに Service Object で明示的に処理できないか

## パフォーマンス

- [ ] N+1クエリが発生していないか
  - [ ] ループ内でアソシエーションを呼び出していないか
  - `includes`、`preload`、`eager_load` の使い分けが適切化
    - `includes`: Rails に任せる（代替手段がある限り絶対に使用しない）
    - `preload`: 基本、`has_many`のような多方向アソシエーションで使用
      - アソシエーション先で絞りたい場合、`joins`と組み合わせて使用
    - `eager_load`: 基本、`has_one`や`belongs_to`などの単方向アソシエーションで使用
- [ ] 大量データをActiveRecordオブジェクトとしてメモリに読み込んでいないか
  - [ ] 特定の属性(idやnameなど)だけ必要な場合は `pluck` を使用しているか
  - [ ] 必要な場合、配列で取得しているか（`to_a` など）
  - [ ] 大量データに対するバッチ処理は `find_each` / `find_in_batches` を使用しているか
- [ ] length / count / size の使い分けが適切か
  - `length`: 未ロードであればロードしてから件数を返すため、ロード済みオブジェクトに対して使用
  - `count`: 常にクエリが発行されるため、ロードされていないオブジェクトに対して使用
  - `size`: ロード済みであればその件数を返し、未ロードであればcountと同様にクエリが発行されるため、両方のケースで使用可能
- [ ] ActiveRecord::Relationに対するデータの存在チェックで`exists?` を使用しているか
  - `any?` や `empty?`や`present?`や`blank?` は対象全件をメモリにロードしてから判定するため、存在チェックには不適切
- [ ] 配列の検索でハッシュテーブルに変換してO(1)で検索できるようにしているか
  - 例: `users.find { |u| u.id == target_id }` はO(n)だが、`users_by_id = users.index_by(&:id); users_by_id[target_id]` はO(1)
- [ ] トランザクションのスコープが必要最小限か
  - [ ] 外部API呼び出し・ファイルI O・重い処理をトランザクション内に含めていないか（ロングトランザクションはDB接続を占有しデッドロックのリスクを高める）
  - [ ] `ActiveRecord::Base.transaction` のブロックが不必要に大きくなっていないか
- [ ] `LIKE '%keyword%'` のような前方一致でないフルテキスト検索をしていないか
- [ ] offset ページネーションとカーソルページネーションの使い分けが適切か
  - offset は後半ページほどパフォーマンスが劣化する（全件スキャン後に読み飛ばすため）
    - ページ番号指定でアクセスしたい管理画面などは offset が許容される
  - 大規模データ・公開APIにはカーソルページネーション（`id > ?` など）を使う

## バグ / ロジックエラー

- [ ] ゼロ除算が発生しうる箇所を処理しているか（割り算の前に除数が `0` でないことを確認）
- [ ] `nil` に対してメソッドを呼び出していないか（`NoMethodError: undefined method for nil`）
  - [ ] `&.` （safe navigation operator）や事前の `nil` チェックで防いでいるか
- [ ] 空配列・空文字列に対して `.first` / `.last` / `[]` でアクセスする場合に `nil` を考慮しているか
- [ ] 整数演算でオーバーフローが起きうる値域を扱っていないか
- [ ] 日付でタイムゾーンが考慮されているか（`Time.now` ではなく `Time.current` / `Time.zone.now`）
- [ ] 日付の大小比較で閏年・月末の境界ケースを考慮しているか
- [ ] 文字列 / 数値の型変換で想定外の入力を考慮しているか
  - [ ] `to_i` は変換失敗時に `0` を返すため、意図しない `0` が混入しないか
  - [ ] `Integer()` など厳格な変換を使うべき箇所はないか
- [ ] 外部入力（params）に想定外の型・値が来た場合の挙動を確認しているか
- [ ] 一覧取得系のAPIで並び順（`order`）が明示的に指定されているか
  - 未指定の場合DBの実装依存になり、ページネーションや表示順が不安定になる
- [ ] レースコンディションを考慮しているか
  - [ ] 複数リクエスト・ジョブが同じレコードを同時に更新する可能性はないか
  - [ ] 楽観的ロック（`lock_version`）または悲観的ロック（`with_lock`）で競合を防いでいるか
  - [ ] `find_or_create_by` はアトミックでないため、一意制約 + リトライ or `upsert` を検討しているか
- [ ] Sidekiq などの非同期ジョブが冪等な実装になっているか
  - Sidekiqはat-least-onceの実行保証のため、同じジョブが重複実行される可能性がある
  - [ ] 重複実行されても副作用が1回になるよう設計されているか（外部API呼び出し・メール送信・レコード更新など）
  - [ ] 冪等でない処理には一意キーによる実行済みチェックを設けているか
- [ ] 浮動小数点の丸め誤差が問題になる箇所はないか
  - [ ] 金額・税率など精度が必要な計算で `Float` を使っていないか（`BigDecimal` を使う）
- [ ] クラスインスタンス変数・クラス変数に副作用を持つ処理を書いていないか
  - Railsはリクエスト間でクラスを再利用するため、クラスレベルの状態変更はスレッドセーフでない
  - `@@variable` や `@class_level_var` に書き込む処理はリクエストをまたいで汚染される可能性がある
- [ ] 文字列が適切に処理されているか
  - Unicode 正規化（NFC/NFD）が統一されていないと、見た目が同じでも文字列比較が一致しないことがある
  - 絵文字・サロゲートペアを含む文字列で `length` がコードポイント数と一致しないことがある（`mb_chars` の使用を検討）
  - [ ] 全角/半角の混在でバリデーションをすり抜けないか
  - Z[ ] IP ファイルをユーザーに返す場合、ファイル名の UTF-8 フラグ（general purpose bit flag の bit 11）を立てているか
    - 未設定だと Windows で展開時にファイル名が文字化けする
    - `rubyzip` では `entry.gp_flags |= 0x0800` で対応

## セキュリティ

- [ ] SQL インジェクションの対策ができているか
  - [ ] 文字列補間でSQLを構築していないか（`where("name = '#{params[:name]}'")`）
    - プレースホルダーを使う：`where("name = ?", params[:name])` または `where(name: params[:name])`
  - [ ] `sanitize_sql` を使わずに生のSQLを組み立てていないか
- [ ] Mass Assignment の対策ができているか
  - [ ] `Strong Parameters` で許可するパラメータを明示的に指定しているか
  - [ ] `permit!` で全パラメータを許可していないか（`permit(:name, :email)` のように明示する）
- [ ] 認可（Authorization）が適切に実装されているか
  - [ ] コントローラーアクションごとに認可チェックがあるか
  - [ ] Pundit / CanCanCan などの認可ライブラリを適切に使っているか
  - [ ] `before_action` で認証・認可をスキップしていないか（`:except` の使いすぎ）
  - [ ] スコープで他ユーザーのリソースにアクセスできないようになっているか
    - `Post.find(params[:id])` ではなく `current_user.posts.find(params[:id])`
  - [ ] 開発中機能に対してフィーチャーフラグが適切に設定されているか（未完成の機能が誤って公開されないように）
- [ ] 危険な文字列が実行されないようにしているか
  - [ ] 外部入力を `eval` に渡していないか
  - [ ] 外部入力を `Open3.capture`・`system`・バッククォートなどのシェルコマンドに渡していないか（コマンドインジェクション）
    - [ ] シェルを介さない配列形式（`Open3.capture2("cmd", arg)`）を使っているか
  - [ ] ファイルパスに外部入力を使っている場合、ディレクトリトラバーサル（`../../etc/passwd` など）を防いでいるか
    - [ ] `File.expand_path` で正規化した上で許可ディレクトリ内に収まることを確認しているか
- [ ] ファイルを扱う際に拡張子やマジックバイトで検証できているか
- [ ] ファイルアップロードで許可する拡張子・MIMEタイプをホワイトリストで管理しているか
- [ ] ファイルサイズの上限をチェックしているか（上限なしだとDoS・ディスク枯渇の原因になる）
- [ ] 脆弱なハッシュ関数を使用していないか
  - [ ] パスワードハッシュ・署名・トークン生成などの暗号用途で `MD5` / `SHA1` を使っていないか
    - 暗号用途には `SHA-256` 以上、パスワードには `bcrypt` / `argon2` を使う
  - [ ] チェックサム・キャッシュキーなど暗号以外の用途であれば `MD5` / `SHA1` は許容される
- [ ] 文字コード・Unicodeの問題を考慮しているか
  - [ ] ホモグリフ（見た目が似た別文字）でフィルターやバリデーションをバイパスできないか
  - [ ] 制御文字・ゼロ幅文字がユーザー入力に混入できないか
- [ ] シークレット管理が適切に行われているか
  - [ ] APIキーや認証情報がコードにハードコードされていないか
  - [ ] `Rails.application.credentials` または環境変数を使っているか
  - [ ] シークレットがログに出力されないか（`filter_parameters` の設定）

## テスト品質

- [ ] `describe` / `context` でテストケースが適切に構造化されているか
- [ ] `let` / `let!` の使い分けが適切か（遅延評価 vs 即時評価）
- [ ] 1つのテスト内でしか使わない値をローカル変数で書けないか（`let` は複数テストで共有する値や `subject` と組み合わせる場合に使う）
  - 不必要な `let` はスコープを広げ、どこで使われているか追いにくくなる
- [ ] `subject` が明確に定義されているか
- [ ] `it`が「正しいこと」「適切であること」のような曖昧な表現になっていないか
  - 具体的な値・状態・メッセージで検証しているか（例: `expect(response).to have_http_status(201)` / `expect(user.name).to eq("Alice")`）
- [ ] フィーチャースペックで JS が必要な場合のみ `js: true` を使っているか
- [ ] 同じ構造で入力値だけ異なるテストが繰り返されていないか
  - [ ] `rspec-parameterized` の `with_them` を使ってパラメータ化できるか
  - [ ] 振る舞いの共通化には `shared_examples` / `it_behaves_like` を使い分けているか
- [ ] `FactoryBot.create` の代わりに `FactoryBot.build` / `build_stubbed` を使えるテストがないか（DBへの書き込みを減らす）
- [ ] ファクトリーにデフォルト値として本番データを使っていないか
- [ ] トレイトを活用して重複を減らしているか
